Getting Started

🔑 Authentication

Every request needs a Bearer token in the Authorization header. Get your token from the KIT Dashboard under Developer settings. Never share it — reset it any time from the same page.

HTTP Header
Authorization: Bearer <your-access-token>
cURL Example
curl -H "Authorization: Bearer <accessToken>" \
     -H "Content-Type: application/json" \
     -X GET https://gateway.kitdashboard.com/payments
🔒
All API calls must use HTTPS. Plain HTTP will fail. Requests without a valid token will return 401 Unauthorized.

⚠️ Errors & Response Codes

Standard HTTP status codes. The response body always includes a machine-readable error structure.

✅
2xx — Success
Request worked as expected
⚠️
4xx — Client Error
Bad input, missing params, auth issues
🔴
5xx — Server Error
Something on our end failed (rare)

Error Response Shape

JSON
{
  "name":    "Unprocessable entity",
  "message": "Amount cannot be blank.",
  "code":    422,
  "status":  422
}

🔍 Filters

Any field marked Filterable can be queried using URL parameters in the format filter[field][operator]=value. Combine operators with and / or.

eq
Equals (=)
neq
Not equals (!=)
lt
Less than (<)
gt
Greater than (>)
lte
Less than or equal
gte
Greater than or equal
like
Contains (LIKE)
in
In a list
nin
Not in a list
or
OR condition
and
AND condition
not
Exclude values

Examples

Created after date
GET /payments?filter[createdOn][gte]=2024-01-01 00:00:00
Date range
GET /payments?filter[and][0][createdOn][gte]=2024-01-01 00:00:00
              &filter[and][0][createdOn][lte]=2024-01-31 23:59:59
Email contains
GET /customers?filter[customer.email][like]=example.com
Multiple IDs
GET /payments?filter[id][in][]=1&filter[id][in][]=2
💡
Use NULL as a value to filter for empty/missing fields. Use filter[not][field]=NULL for non-empty.

Payment Gateway

💳 Payment Gateway

The Gateway API handles all card-present and card-not-present transactions — sales, pre-auths, captures, refunds, and tokenization. Supports Level 1, 2, and 3 processing (L2/L3 unlock lower interchange rates). Sandbox available for testing without real money.

Base URLs
Productionhttps://gateway.kitdashboard.com
Sandboxhttps://sandbox-gateway.kitdashboard.com

List Payments

Returns a paginated list of all payment transactions. Supports full filter syntax on any filterable field.

GET /payments All transactions
Request
curl -H "Authorization: Bearer <accessToken>" \
     https://gateway.kitdashboard.com/payments
With filters
GET /payments?filter[terminal.id][eq]=24&filter[type][eq]=Sale&page=1&per-page=20

Sale

Charge a card in one step — authorization + capture together. Use when you don't need a separate capture step. Supports L1/L2/L3 processing and Soft/Dynamic Descriptors.

POST /payment/sale Authorize + capture

card.nameStringoptionalCardholder name
card.addressObjectoptionalBilling address for AVS check

Request (Level 1)

    "terminal": { "id": 24 },

Level 2 / 3 processing requires additional order-level data (tax amount, PO number, line items). It reduces interchange fees significantly for B2B transactions.

Auth (Pre-Authorization)

Authorize a card without immediately capturing funds. The card is checked and a hold placed. You then capture later. Common for hotels, car rentals, and restaurants.

POST /payment/auth Hold funds, capture later
💡
After an Auth, use Capture to actually collect the funds. Partial capture is supported — you can capture less than the authorized amount.

Auth Response Codes

Error200+Processing error — may be retried

Capture

Capture a previously authorized payment. Can capture the full amount or partial (less than authorized).

POST /payment/{id}/capture Capture authorized funds
POST /payment/{id}/capture/partial Partial capture

Refund

Refund a captured transaction. Full or partial refunds are supported.

POST /payment/{id}/refund Full refund
POST /payment/{id}/refund/partial Partial refund

Card Authentication

Verify a card is real and active without charging it. Returns a verification result — useful for adding cards to a customer vault before the first charge.

POST /payment/card-authentication Verify card without charging

Generate Token

Tokenize a card number for later use. The token can be stored safely and used in place of the raw card number for future charges.

POST /payment/generate-token Tokenize a card

Hosted Form

A pre-built, PCI-compliant payment form hosted by KIT. You pass configuration and get back a form URL or embed code. Customer card data never touches your server.

POST /hosted-form Create a hosted payment form
✅
Simplest PCI path. Use this if you don't want to handle card data at all. The hosted form captures everything and posts the tokenized result to your callback URL.

Hosted Fields

Embeddable iFrame fields for card number, expiry, and CVV. They render inside your UI but live on KIT's domain, so you stay out of PCI scope. Get a token from the fields, then use it to charge.

POST /hosted-fields/token Exchange field data for a payment token
💡
Best for custom UIs. Your checkout looks exactly how you want it, but card numbers never reach your server. Use the returned token in any charge endpoint.

📝 Boarding Applications

The full merchant onboarding workflow. An application belongs to a Campaign (which controls pricing, processing type, fees). The flow is linear — create first, then fill in sections, then submit.

📋
Standard boarding flow:
1. Create application → get ID
2. Update company + DBA data
3. Add/update Principals
4. Fill Processing or ACH section
5. Upload + assign Documents
6. Update Fees if needed
7. Request eSign (or use AutoSign)
8. Set status to Underwriting

Recurring

🔄 Recurring Payments

Schedule automatic payments for a customer on any interval using cron expressions. The system retries failed payments up to 3 times per card, and if the customer has multiple cards on file it tries each one in order.

🔁
Retry logic: Up to 3 attempts per card. If a customer has multiple cards and no specific card is set, each card is tried in the order saved in their wallet.

customer.idIntegerCustomer in the Vaultrequired
customer.card.idIntegerSpecific card to use (optional)optional

Create recurring — cURL

  -H "Authorization: Bearer <accessToken>" \
  -H "Content-Type: application/json" \

    "amount": 29.99,

    "terminal": { "id": 47 },
    "customer": { "id": 1, "card": { "id": 2 } }
  }'

🛡️ 3D Secure

3DS adds an extra authentication step (like a one-time code from the card's bank) before authorizing a payment. It shifts liability for fraud from you to the card issuer when used.

POST/3ds/createInitiate 3DS session
POST/3ds/{id}/checkCheck 3DS status
FieldRequiredDescription
terminal.idrequiredDashboard Terminal ID
amountrequiredTransaction amount
card.numberrequiredFull card number
card.exprequiredExpiry MM/YY
Create 3DS session
curl -X POST https://gateway.kitdashboard.com/3ds/create \
  -H "Authorization: Bearer <accessToken>" \
  -H "Content-Type: application/json" \
  -d '{
    "terminal": { "id": 1 },
    "amount": "45.00",
    "card": { "number": 5555555555554444, "exp": "12/25" }
  }'
// Returns: { "id": "1" } — use this id to poll /3ds/1/check

🔔 Webhooks

Subscribe to real-time event notifications. When something happens in the system (new transaction, chargeback, status change), KIT POSTs a JSON payload to your endpoint. Configure webhook URLs and signing secrets in the Dashboard.

Webhook Object

Base structure
{
  "id":     51,              // Event ID
  "module": "transaction",   // Event group
  "action": "create",        // Specific event
  "date":   "2025-11-17 15:01:03",
  "data":   { /* Event payload */ }
}

Signature Verification

Every webhook includes a Webhook-Signature header — a SHA-512 hash of webhookSignature + id + module + action + date. Always verify it to confirm the event came from KIT POS.

Verify in Node.js
const crypto = require('crypto');
const expected = crypto
  .createHash('sha512')
  .update(secret + id + module + action + date)
  .digest('hex');
const valid = expected === req.headers['webhook-signature'];
